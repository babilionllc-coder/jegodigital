# HARD RULE #16 — Never Enable Email Tracking on Cold Campaigns

> **Last updated:** 2026-05-01
> **Status:** ACTIVE — enforced by `tools/verify_no_tracking.sh` (live API check)
> **Owner:** Alex Jego — directive given 2026-05-01 PM after live diagnosis of 0% open rate disaster

---

## 🛑 The rule (single sentence)

**Every cold-email campaign in Instantly MUST ship with `link_tracking: false` AND `open_tracking: false`. No exceptions, no A/B tests, no "just for this one campaign".**

---

## Why (the deliverability research)

In 2026 Gmail's spam classifier downranks any email that:

1. **Loads a tracking pixel** (used to measure opens) — Apple Mail Privacy Protection (MPP) since 2022 + Gmail's privacy filter since 2024 silently treat pixel-loading emails as low-quality. Pixel-loaded emails get ~15-30% lower Primary inbox placement.
2. **Rewrites links through a redirect domain** (used to measure clicks) — visible link text doesn't match actual URL = classic phishing pattern. Even if the redirect domain is yours, Gmail flags the mismatch.
3. **Routes clicks through a low-reputation subdomain** — fresh tracking subdomains like `inst.zennoenigmawire.com` (the JegoDigital CTD that shipped 2026-04-22) have near-zero reputation, so Gmail treats them as suspicious redirectors.

Top-performing senders (Smartlead/lemlist/Instantly elite tier) ALL ship with both flags off and rely purely on **reply rate** as the only honest cold-email metric.

---

## What this means in practice

| Setting | Required value | Reason |
|---|---|---|
| `open_tracking` | `false` | Removes the 1×1 pixel from the email body |
| `link_tracking` | `false` | Stops Instantly from rewriting URLs through the CTD |
| Custom Tracking Domain (CTD) | NOT configured per-campaign | Even with flags off, a stale CTD reference can leak into headers |
| Reply rate | THE primary metric | Replace any open-rate dashboard with reply-rate |

**Forbidden:**
- ❌ Suggesting "fix the tracking domain" as a deliverability remedy — that IS the deliverability problem
- ❌ Diagnosing 0% open rate as a bug — it's intentional and correct
- ❌ Recommending CTD setup as part of any "clean campaign" checklist
- ❌ Comparing JegoDigital's open-rate to industry benchmarks — incomparable, ours is intentionally untracked
- ❌ Building reports/dashboards that use open or click data as a primary KPI

**Required:**
- ✅ Every new campaign created via API MUST be created with `{"link_tracking": false, "open_tracking": false}` in the POST body
- ✅ Pre-launch validator (`tools/verify_no_tracking.sh`) MUST pass before activating any campaign
- ✅ Monthly cron sweep flags any campaign that drifts back to tracking=on (no UI default change should be allowed to silently re-enable)

## What the disaster looked like (2026-05-01)

**Symptom:** 224 emails sent across 3 active USA campaigns over 30 days. **0 opens recorded, 0 replies.** Bounce rate normal (3%).

**Initial wrong diagnosis (Claude, before Alex shut it down):** "Tracking domain is broken — fix `inst.zennoenigmawire.com` to restore open data flow."

**Correct diagnosis (after Alex's pushback + live API audit):**
- `open_tracking: false` ✅ (Alex correctly disabled)
- `link_tracking: true` 🚨 on **8 of 22 campaigns** (residual default)
- Even with open pixel off, every URL was being rewritten through the Vercel-hosted CTD `inst.zennoenigmawire.com` (66.33.60.193 + 76.76.21.241)
- Gmail saw fresh-subdomain redirects + visible-link-text-mismatch = spam folder
- The "0% open rate" was actually a feature (no pixel = no measurement) but emails were still being downranked because of the link rewrite

**Fix shipped 2026-05-01 PM:**
- 8 campaigns PATCHed `link_tracking: false` via Instantly v2 API
- 1 additional campaign PATCHed `open_tracking: false`
- Final state verified: 0 / 22 campaigns with any tracking enabled
- HR-16 added to CLAUDE.md
- `tools/verify_no_tracking.sh` shipped as pre-launch validator
- This file exists so it never recurs
