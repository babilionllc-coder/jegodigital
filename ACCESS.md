## HOW KEYS FLOW TO PRODUCTION (Read this ONCE, never forget)

```
┌─────────────────────┐      ┌──────────────────────┐      ┌──────────────────┐
│ GitHub Secrets (28) │ ───▶ │ deploy.yml heredoc   │ ───▶ │ functions/.env   │
│ Source of truth     │      │ Must name each key   │      │ Live Cloud Fns   │
│ github.com/.../     │      │ .github/workflows/   │      │ (auto-generated  │
│ settings/secrets    │      │ deploy.yml           │      │ on every deploy) │
└─────────────────────┘      └──────────────────────┘      └──────────────────┘
```

**THE RULE:** A key reaches production ONLY when (a) it's in GitHub Secrets AND (b) its name is in the heredoc in `.github/workflows/deploy.yml`. Missing either = stranded.

**WHY WE HAD MISSING KEYS:** The heredoc is a hardcoded whitelist. Every new key needs two adds (Secrets + heredoc). Nobody documented this.

**THE FIX:** This file + the `envAudit` cron that alerts Slack if any required key is missing at runtime.

## HOW TO ADD A NEW KEY (checklist)

1. **Add to GitHub Secrets:** `https://github.com/babilionllc-coder/jegodigital/settings/secrets/actions`
2. **Add line to `deploy.yml` heredoc:** `KEY_NAME=${{ secrets.KEY_NAME }}`
3. **Add row to this file's registry** with who uses it + rotation cadence
4. **Add to `envAudit.js` REQUIRED_KEYS list** if the key is production-critical
5. **Push a tiny change to `website/**`** (or to `.github/workflows/deploy.yml`) so the deploy fires
6. **Verify:** in next day's 6am Slack digest, `envAudit` reports "all keys present ✅"

## HOW TO ROTATE A KEY

1. Generate new key at provider (Brevo, Instantly, etc.)
2. Update GitHub Secret (same name, new value)
3. Bump commit (e.g., touch a workflow file) → deploy fires → new `functions/.env` has new value
4. Deprecate the old key at provider
5. Update this file's "Last verified" column
